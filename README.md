## 🔍 How I Think About Support Problems

Good support is not about answering fast — it's about answering right. My approach follows a consistent logic:

```
1. UNDERSTAND  →  What exactly is the user experiencing?
2. REPRODUCE   →  Can I see the same behavior?
3. INVESTIGATE →  What do the logs, data, or system state tell me?
4. ISOLATE     →  Is it a user error, a configuration issue, or a bug?
5. RESOLVE     →  Fix it, work around it, or escalate with full context.
6. COMMUNICATE →  Keep the user informed. Always close the loop.
```

This mindset is reflected in every case study and SQL query in this portfolio.

## 📊 Why SQL for Support?

In modern SaaS companies, support analysts often have direct or read-only access to databases or data tools. Being able to run a query like:

> *"Show me all tickets opened in the last 7 days where the user's plan is Pro and the issue category is 'Login'"*

...saves hours compared to manually exporting spreadsheets. The SQL files in this repo reflect the kinds of questions I ask when investigating issues or preparing reports for weekly syncs.
